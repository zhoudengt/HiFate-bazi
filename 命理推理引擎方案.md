# 命理推理引擎方案 — 感情婚姻领域起步

> 创建日期：2026-03-18
> 关联文档：`AI质量提升路线图.md`、`docs/感情婚姻_系统提示词_v2.md`

---

## 一、核心思路

在现有**计算层**（排盘/旺衰/十神/神煞）与**LLM**之间，插入一层**确定性推理引擎**。

- 计算层：已有，负责精确排盘
- **推理层（新增）**：从计算结果出发，通过规则化因果链条推导出具体结论
- LLM层：只负责将推理结论写成自然语言，不再自行推理

推理引擎失败时自动降级到现有行为（仅传数据），前端 SSE 格式完全不变。

---

## 二、架构

```
用户输入 → BaziDataOrchestrator(现有)
         ├→ RuleService.match() → matched_rules（编排器已匹配的 bazi_rules）
         └→ build_input_data(现有)
              → engine.build_input(... matched_rules=matched_rules)
              → InferenceEngine.infer(新增，安全降级)
                 ├→ 代码内置推理（spouse_star / marriage_palace / dynamic_balance / timing）
                 ├→ _infer_from_db_rules()  ← MarriageMatchContext + InferenceConditionMatcher 严格匹配
                 │   └→ bazi_stream_inference_rules 表（103条婚姻规则，43个条件键全注册）
                 └→ _infer_from_bazi_rules() ← matched_rules（编排器传入，零额外查询）
              → InferenceQualityFilter.apply()（四层过滤：置信度/矛盾/去重/限数）
              → format_with_inference(分层标记：★★核心 / ★重点 / 参考)
              → LLM 流式输出(现有)
```

### 双规则来源

| 来源 | 表名 | 匹配方式 | 特点 |
|------|------|----------|------|
| 编排器匹配 | `bazi_rules` | `EnhancedRuleCondition.match()` | 覆盖面广，文本判词，已有 400+ 条婚姻规则 |
| 推理引擎自有 | `bazi_stream_inference_rules` | `InferenceConditionMatcher 严格匹配` | 多因子因果链，43个条件键全注册，未知键→不匹配 |

### 共用组件（所有领域复用）

- `core/inference/models.py` — 通用数据模型（InferenceInput 含 matched_bazi_rules）
- `core/inference/base_engine.py` — 推理引擎抽象基类（单例 + 降级 + build_input 含 matched_rules）
- `core/inference/condition_matcher.py` — **条件匹配框架**（BaseMatchContext + 注册表 + 严格 match）
- `core/inference/condition_checkers/common.py` — **29 个通用条件检查器**（性别/旺衰/十神/神煞/地支/天干/格局/日柱）
- `core/inference/condition_checkers/{domain}.py` — **领域特有检查器**（如 marriage.py 12个 + MarriageMatchContext）
- `core/inference/quality_filter.py` — **四层质量过滤器**（置信度/矛盾/去重/限数）
- `core/inference/force_balancer.py` — 动态力场平衡器（含合化/冲散/天干合化逻辑）
- `core/inference/rule_loader.py` — MySQL 规则加载器（缓存 + 热更新）
- `server/utils/prompts/inference_formatter.py` — 推理结论格式化器（分层标记 ★★/★，核心/参考分组）

### 开发规范

> 详见 `.cursor/rules/inference-engine.mdc`，新增领域严格按 9 步标准流程执行。

### 婚姻引擎 6 个推理维度

1. **配偶星分析（spouse_star）**：男看财星，女看官杀，判断配偶特质与婚姻质量
2. **婚姻宫分析（marriage_palace）**：日支十神含义 + 冲合刑害 + 神煞
3. **大运动态力场（dynamic_balance）**：每步大运加入后五行力量重算（含合化/冲散）
4. **婚恋时机（marriage_timing）**：大运引动配偶星/婚姻宫 + 特殊流年
5. **DB 推理规则（db_rule）**：从 bazi_stream_inference_rules 表匹配多因子规则
6. **bazi_rules 整合（bazi_rule_judgment）**：从编排器已匹配的 411 条规则中提取婚姻判词

### 力场计算升级（P1）

| 特性 | V1 | V2 |
|------|-----|-----|
| 基础力量 | 天干+地支+藏干加法 | 同左 |
| 冲的影响 | 仅记录关系 | **被冲地支力量 -0.8** |
| 合的影响 | 仅记录关系 | **合化五行力量 +0.6** |
| 刑的影响 | 仅记录关系 | **受刑地支力量 -0.5** |
| 天干合化 | 无 | **天干合化五行力量 +0.5** |
| 五行下限 | 可能为负数 | **`max(0, ...)` 保底** |

### 置信度可调参数（P2）

`MarriageInferenceEngine.CONFIDENCE_BASE` 字典存储每种 `rule_type` 的基础置信度。可通过 `update_confidence_base()` 运行时调参：

```python
MarriageInferenceEngine.update_confidence_base({
    'marriage_day_pillar': 0.85,
    'marriage_ten_gods': 0.80,
})
```

## 八、质量规范（所有领域必须遵守）

> 婚姻领域开发中踩过的坑，已修复并沉淀为规范。后续扩展健康/事业等领域时，**必须逐条检查**。

### 8.1 格式化输出禁止 raw dict

`branch_relations` 等字段可能是 dict/list/str 多种格式。格式化给 LLM 前必须转为中文。

**做法：** 所有引擎必须实现 `_format_branch_pair(pair)` 工具方法，处理 dict（取 `branches` 字段拼接）、list（直接拼接）、str（原样返回）三种情况。

**反面案例：** `婚姻宫戌被冲（{'pillars': ['year', 'day'], 'branches': ['辰', '戌']}）` → LLM 收到脏数据。
**正面案例：** `婚姻宫戌被冲（辰戌）`

### 8.2 柱位名称必须中文

代码内部用 `year/month/day/hour`，但输出给 LLM 必须翻译为 `年柱/月柱/日柱/时柱`。

**做法：** 每个引擎文件顶部定义 `PILLAR_CN = {'year': '年柱', 'month': '月柱', 'day': '日柱', 'hour': '时柱'}`，输出时统一用 `PILLAR_CN.get(pillar, pillar)`。

### 8.3 推理链条必须去重

代码内置推理和 DB 规则可能对同一个命局特征产出相似结论。必须在 `_run_inference()` 最后调用 `_deduplicate_chains()`。

**去重逻辑：** 同 category 下，取 conclusion 前 20 字符作为相似度 key，保留 confidence 最高的那条。

### 8.4 年龄范围必须按领域过滤

不同领域关注的年龄段不同，大运力场计算和婚恋时机推理都要用 `age_range` 参数过滤。

| 领域 | age_range | 说明 |
|------|-----------|------|
| 婚姻 | (18, 50) | 婚恋适龄 |
| 事业 | (22, 60) | 职业生涯 |
| 健康 | (0, 90) | 全生命周期 |
| 子女 | (20, 50) | 生育适龄 |
| 年运 | 当前年龄±5 | 近期关注 |

`force_balancer.calculate_all_dayun_balances()` 已支持 `age_range` 参数，过滤逻辑：大运起始年龄超过上限 **或** 结束年龄不到下限的，跳过。

### 8.5 DB 规则 conditions 不可为空白匹配

`conditions: {}` 的规则（如「天合地合大吉年」）会无条件匹配所有命局。必须加约束：

**做法：** 在 `_infer_from_db_rules()` 中，当 `conditions` 为空且 `category == 'marriage_timing'` 时，检查命局是否确实有特殊流年数据，没有则跳过。

**通用原则：** 每个领域的 DB 规则，空 conditions 必须配合 category 做二次校验，避免无差别匹配。

### 8.6 格式化器必须输出来源和可信度

LLM 需要知道哪些结论更可信、来自哪本经典，以便在润色时区分主次。

**格式规范：**
```
条件，→ 结论（时段，据《来源》，可信度高/仅供参考）
```

| confidence | 标注 |
|------------|------|
| ≥ 0.85 | `可信度高` |
| 0.70 ~ 0.84 | 不标注（默认） |
| < 0.70 | `仅供参考` |

### 8.7 SQL 文件必须带 SET NAMES utf8mb4

生产 MySQL 默认连接字符集是 `latin1`，不加 `SET NAMES utf8mb4` 会导致中文双重编码变乱码。

**做法：** 每个 SQL 文件第一行必须是 `SET NAMES utf8mb4;`。导入命令也加 `--default-character-set=utf8mb4`。

### 8.8 种子规则 INSERT 必须幂等

使用 `ON DUPLICATE KEY UPDATE` 确保重复执行不会重复插入，只更新已有记录。`rule_code` 作为唯一键。

### 8.9 利用 bazi_rules 已匹配规则

编排器通过 `EnhancedRuleCondition.match()` 已匹配的 `bazi_rules` 规则（`matched_rules`），无需在推理引擎中重复匹配。

**做法：**
1. `build_input()` 接收 `matched_rules` 参数
2. `InferenceInput.matched_bazi_rules` 存储已匹配规则
3. 引擎中新增 `_infer_from_bazi_rules()` 方法，从 matched_rules 中提取本领域判词
4. 使用 `BAZI_RULE_CATEGORY_MAP` 映射 `rule_type` → 推理类别

**效果：** 零额外查询开销，直接复用编排器已匹配的数百条规则。

### 8.10 力场计算必须包含合化/冲散

纯加法力场不够精确。大运干支与原局的合化、冲散会显著改变五行力量。

**做法：**
- 冲：被冲地支力量 -0.8
- 六合化：合化后五行力量 +0.6
- 刑：受刑地支力量 -0.5
- 天干合化：合化五行力量 +0.5
- 五行力量下限为 0（`max(0, ...)`）

### 8.11 置信度必须可调参数化

不同 `rule_type` 的规则可靠性不同。必须通过 `CONFIDENCE_BASE` 字典管理基础置信度，而非硬编码。

**做法：**
- 每个领域引擎定义 `CONFIDENCE_BASE` 类变量
- 提供 `update_confidence_base(overrides)` 类方法支持运行时调参
- 优先级 bonus：`priority_bonus = max(0, (priority - 100)) * 0.001`
- 上限 0.95

### 8.12 测试集必须覆盖所有推理维度

每个领域至少 20 个测试用例，覆盖：
- 基础推理（配偶星/脏腑/事业星等）
- 大运力场
- bazi_rules 整合
- 降级（空数据不崩溃）
- 多规则混合
- 置信度分层
